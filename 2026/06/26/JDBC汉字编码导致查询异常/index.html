诡异的是，同事在 Navicat 里用同样的条件一查，明明白白 2 条记录躺在那里。

这不是第一次遇到这种问题了，但我决定这次彻底搞清楚。

二、第一轮排查：怀疑锁

看到 FOR UPDATE SKIP LOCKED，我第一反应是锁的问题。

SKIP LOCKED 的语义是：如果目标行已经被其他事务加了排他锁，直接跳过，不等待、不报错。所以当前事务看到的可能是 0 条。

我在 Navicat 里原封不动跑了同样的 SQL：

结果：0 条。 去掉 FOR UPDATE SKIP LOCKED 再跑：2 条。

锁确实是表象，但这不是故事的终点——为什么我的 Navicat 能查到 2 条，而代码只返回 0 条？

因为我在排查时用的是「去掉锁」的 Navicat 查询，而代码永远带着 SKIP LOCKED。真正的问题是：为什么这 2 行一上来就被锁了？谁锁的？

三、第二轮排查：缩小范围

接下来的思路是分而治之，逐个排除：

怀疑方向 验证方法 结果
is_valid IS NULL vs is_valid = 0 直接 SELECT 看字段值 ✅ 值是 NULL，条件正确
BigDecimal 精度问题 50.00 vs 50 ✅ MySQL 自动转换，没问题
事务隔离级别 已提交/未提交数据 ✅ 都是已提交数据
currency_type = '泰铢' —— 等等… 把条件换成 id = xxx 直接定位

我把查询条件从 currency_type = '泰铢' 换成了这 2 条记录的具体 id：

罪魁祸首浮出水面：currency_type = '泰铢' 这个条件。

确切地说，不是 SQL 写错了，而是 Java 传到 MySQL 的那两个汉字，已经不是 泰铢 了。

四、根因：字符编码的「隐身术」

画一张数据流图就很清楚了：

问题的根源就在 JDBC 连接 URL 上：

MySQL Connector/J 在没有显式指定 characterEncoding 时，默认使用 latin1（ISO-8859-1）作为客户端字符集。latin1 是单字节编码，只支持 0-255 的字符。任何超出这个范围的字符——包括所有中文字符——都会被”错误转码”。

详细过程是这样的：

  1. Java 字符串 "泰铢" 在内存中是以 UTF-16 存储的
  2. JDBC 驱动按照连接字符集 latin1 将字符串编码成字节流
  3. UTF-16 → latin1 的转换是破坏性的：每个中文字符（2 字节）被拆成 2 个 latin1 字符
  4. MySQL 服务端收到后，按自己的 character_set_client 再解码，得到的结果已经不是 泰铢，而是一串乱码
  5. WHERE currency_type = 乱码 自然什么都匹配不到

五、为什么数字条件不受影响？

这是一个关键细节，也是这个问题容易被忽略的原因。

ASCII 字符（0-9、a-z、A-Z）在所有字符集中的字节表示完全一致。 数字 50 无论用 latin1 还是 utf8mb4 传输，字节都是 0x35 0x30，不会有任何歧义。

只有非 ASCII 字符——中文、日文、韩文、Emoji 等——才会在不同字符集之间出现”翻译错误”。

六、修复

在 JDBC URL 上加上三个参数，问题解决：

参数 含义 必要性
useUnicode=true 启用 Unicode 字符集支持 必须
characterEncoding=utf8mb4 客户端发送数据的编码 必须，用 utf8mb4 而非 utf8
characterSetResults=utf8mb4 结果集返回的编码 建议加上

⚠️ 为什么用 utf8mb4 而不是 utf8？

MySQL 的 utf8 是阉割版的 UTF-8，最多只支持 3 字节字符（无法存储 Emoji 和部分生僻汉字）。utf8mb4 才是完整的 UTF-8 实现，支持 4 字节字符。永远用 utf8mb4。

八、复盘总结

排查阶段 走过的弯路 正确思路
第一反应 怀疑 SKIP LOCKED 锁问题 ✅ 确实是锁，但那是因为别的查询也因编码问题没 commit
排查 is_valid 怀疑 NULL vs 0 ❌ 方向错了，字段值就是 NULL
定位汉字 终于发现关键线索 ✅ 把中文条件换成 id 对比
发现根因 — ✅ JDBC URL 缺 characterEncoding

一句话总结：JDBC 连接没有指定 characterEncoding=utf8mb4，导致中文字符在传输过程中被 latin1 错误转码，"泰铢" 变成了乱码，WHERE 条件自然匹配不到任何数据。

九、延伸：一个自查清单

如果你的项目中也有类似「Navicat 能查到、代码查不到」的问题，按这个顺序排查：

  1. 先别怀疑锁 —— 虽然 SKIP LOCKED 经常背锅，但先排除更基础的问题
  2. 把中文条件换成数字 id 对比 —— 如果 id 能查到、中文查不到，就是编码问题
  3. 检查 JDBC URL 里的 characterEncoding —— 90% 的汉字查询异常都是这个原因
  4. 检查 MySQL 全局编码 —— SHOW VARIABLES LIKE 'character_set_%'
  5. 检查表/字段级别的编码 —— SHOW FULL COLUMNS FROM your_table
  6. 以上都排除了，再回头排查事务、锁、隔离级别
