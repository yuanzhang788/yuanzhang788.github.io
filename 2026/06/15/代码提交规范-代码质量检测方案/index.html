代码提交规范 & 代码质量检测方案

发表于2026-06-15|更新于2026-06-15
|浏览量:

适用项目：

  • 基于 prod / test / dev-feature-xx 三分支模型代码托管
  • GiteeCI/CD：Jenkins / 阿里云云效
  • 代码质量：SonarQube/阿里云云效代码检测

一、分支模型

代码分支模型html(需启用脚本)

Git 提交节点流转图

目前代码分支

prod(主分支) test(测试分支) dev-feature-xx(开发分支)

  1. 锁主分支分支不许提交
  2. 每次开发新需求从从prod 拉取最新的分支作为自己的开发分支
  3. 开发完成后(合并到test分支)进行测试
    1. 向发起merge request(如需审查)
    2. 直接向test分支合并(不需审查)
  4. 测试完成后
    1. 如有问题 继续回到自己的开发分支修改问题后再次合并到test分支进行测试
    2. 如没有问题 进行步骤5
  5. 步骤5 将自己对应的开发分支通过merge request合并到prod (代码审查通过后合并至prod),进行发版

发起MR以及代码审查

发起MR

进行代码审查

分支 用途 保护规则
prod 生产环境代码，始终保持可发布状态 🔒 禁止直接推送，仅通过 MR 合入
test 测试环境代码，用于集成测试 允许 MR 合入或直接合入
dev-feature-xx 功能开发分支，每人/每需求独立 无限制，开发者自行管理

三、代码质量分析方案

提供 三种方案，按推荐程度排序：

方案 工具 推荐度 特点
方案一 阿里云云效流水线（推荐） ⭐⭐⭐⭐⭐ 开箱即用，无需自建，与代码托管深度集成
方案二 阿里云云效代码检测（独立） ⭐⭐⭐⭐ 提交时自动检测，轻量灵活
方案三 Jenkins + SonarQube（自建） ⭐⭐⭐ 灵活度最高，需自行搭建维护

方案一：阿里云云效流水线进行代码检测（⭐ 推荐）

云效流水线内置代码扫描能力，与代码库、分支规则无缝集成，无需单独搭建 SonarQube。

1. 创建流水线

进入 云效 → 流水线 → 新建流水线，选择「空白流水线」或「Java / Node.js 模板」。

2. 配置流水线阶段

代码检出 ──→ 代码扫描 ──→ 单元测试 ──→ 构建 ──→ 部署

阶段一：代码检出
  • 源码来源：选择 Gitee 代码库
  • 分支：配置为触发分支（test / prod）
  • 触发方式：
    • 合并到 test → 触发测试流水线
    • 合并到 prod → 触发生产流水线
阶段二：代码扫描（SonarQube 内置）

添加任务「代码扫描」，云效内置了基于 SonarQube 引擎的扫描能力：

  2. 点击「创建检测任务」

四、三种方案对比

对比维度 云效流水线（推荐） 云效代码检测（独立） Jenkins + SonarQube（自建）
搭建成本 零成本，开箱即用 零成本，开箱即用 需搭建 Jenkins + SonarQube
维护成本 云效托管，免维护 云效托管，免维护 需自行维护服务器和版本升级
集成度 与云效代码库深度集成 与代码提交/MR 集成 需手动配置 Webhook 和插件
扫描能力 基于 SonarQube 引擎 基于 SonarQube 引擎 完整 SonarQube 功能
自定义规则 支持自定义 支持自定义 完全自定义
Quality Gate ✅ 支持 ✅ 支持 ✅ 支持
流水线集成 ✅ 原生集成 需配合流水线使用 ✅ 原生集成
多分支分析 ✅ 支持 ✅ 支持 ✅ 支持
适合场景 已使用或计划使用云效 只需代码检测，不需完整流水线 已有 Jenkins 基础设施，追求完全自主

五、推荐实施路径

如果项目已在使用云效 → 直接用方案一，零成本接入如果项目已有 Jenkins → 用方案三，在现有基础上加 SonarQube如果只想快速看代码质量 → 用方案二，5 分钟完成接入

六、Quality Gate 推荐规则

指标 测试线（宽松） 生产线（严格）
新增 Blocker = 0 = 0
新增 Critical = 0 = 0
新增代码覆盖率 ≥ 60% ≥ 80%
整体覆盖率 ≥ 50% ≥ 80%
新增代码重复率 < 5% < 3%
安全热点审查率 > 80% 100%
可靠性评级 ≥ A ≥ A
安全评级 ≥ A ≥ A
可维护性评级 ≥ B ≥ A

💡 建议：初次接入时使用测试线规则（宽松），运行 1-2 周后逐步收紧至生产线规则，避免一次性阻断大量流水线。
