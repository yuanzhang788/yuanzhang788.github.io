  1. 因为之前系统都是正常的,我想到的就是会不是硬盘满了
    1. 使用如下命令查看服务器磁盘使用情况(发现docker overlay2占用100%导致的)

![](/assets/问题排查记录(清理 Docker 无用资源)/de5c643595e40a5f71004a6b5234c0fb.png)
3. docker查看占用

![](/assets/问题排查记录(清理 Docker 无用资源)/f9e11ddcda26ecf5ab66d2952564192c.png)

docker system df 输出，问题很明显：

  • Images：16 个镜像，占用 6.7GB，可回收 829MB
  • Containers：14 个容器，占用 477MB
  • Build Cache：292 个缓存，占用 66.87GB，可全部回收

也就是说，根分区 / 被 Docker 的 Build Cache 和 Overlay2 文件占满了，这正是导致 Redis、Kafka 报错的原因。

⚠Docker 资源清理Docker

清理 Docker 无用资源命令,因为我这里是Build Cache占用过高,所以使用的是清除缓存的命令

  • docker system prune -a -f
    • 执行后，会强制清理掉：
    • 停止的容器
    • 不再使用的镜像
    • 不再使用的网络
    • 构建缓存
  • docker volume prune -f
    • 清理 所有未被容器使用的卷（volumes）。
  • docker builder prune -a -f
    • 一个专门用来清理 Docker 在构建镜像时生成的中间层和缓存（Build Cache） 的命令。 它比 docker system prune 更精准地聚焦在构建层面。

🔷区别对比

对比项 builder prune -f builder prune -a -f system prune -a -f
删除目标 构建缓存（安全部分） 所有构建缓存 整个 Docker 系统中未使用的资源
删除镜像 ❌ ❌ ✅（未使用的）
删除容器 ❌ ❌ ✅（已停止的）
删除网络 ❌ ❌ ✅（未使用的）
删除卷 ❌ ❌ ✅（仅带 --volumes
时）
删除构建缓存 ✅（未使用） ✅（全部） ✅
空间释放量 ⭐ ⭐⭐⭐ ⭐⭐⭐⭐⭐
风险 低 中 高

👉建议使用策略

场景 推荐命令 原因
日常开发清理缓存 docker builder prune -f 安全，不会影响构建
构建服务器空间紧张 docker builder prune -a -f 释放缓存层，构建慢一点但安全
整个 Docker 环境空间爆满 docker system prune -a -f 全面清理，代价大但最彻底

✅ 一句话总结：

命令 一句话总结
docker builder prune -f 清理构建缓存中“安全可删”的层
docker builder prune -a -f 清空所有构建缓存层，重新构建更慢但释放大空间
docker system prune -a -f 清空整个 Docker 环境未使用的资源，最彻底也最危险

✅本次命令选择

因为我们只清理缓存所以使用docker builder prune相关命令

因为我们的镜像后续还会使用,所以选择不清除全部,选择清除未使用的构建缓存,因此选用docker builder prune -f命令

![](/assets/问题排查记录(清理 Docker 无用资源)/83107696e23b3752165b894eb900642d.png)

清理完成效果,后续再看看哪些服务挂掉了,重启一下就完事了
